│   └─ Monitoring

│   💡 Các task tương tự PM:

│   │

#Google Project Management – Waterfall vs Agile Project Management Methodologies

```diagram
Waterfall vs Agile Project Management Methodologies
│
├─ 1. Tổng quan ⭐
│   ├─ 2 phương pháp phổ biến:
│   │   ├─ Waterfall (truyền thống, tuyến tính)
│   │   └─ Agile (linh hoạt, lặp lại)
│   ├─ Mục tiêu:
│   │   └─ Giúp tổ chức dự án hiệu quả hơn tùy theo bối cảnh
│   └─ Cả hai đều dựa trên Project Life Cycle (Initiate → Plan → Execute → Close)
│
├─ 2. Waterfall Methodology (Mô hình thác nước) ⭐
│
│   ├─ Khái niệm:
│   │   ├─ Sequential (tuần tự)
│   │   ├─ Linear (tuyến tính)
│   │   └─ Làm từng bước theo thứ tự cố định
│   │
│   ├─ Đặc điểm:
│   │   ├─ Phases rõ ràng, không chồng lấn
│   │   ├─ Ít thay đổi sau khi bắt đầu
│   │   └─ Yêu cầu kế hoạch chi tiết từ đầu
│   │
│   ├─ Khi nào dùng Waterfall:
│   │   ├─ Requirements rõ ràng ngay từ đầu
│   │   ├─ Task phụ thuộc lẫn nhau (phải xong bước trước mới làm bước sau)
│   │   ├─ Thay đổi rất tốn kém
│   │   └─ Dự án ổn định, ít rủi ro thay đổi
│   │
│   ├─ Ví dụ:
│   │   ├─ Catering event (tổ chức tiệc)
│   │   ├─ Xác định số khách → menu → đặt nguyên liệu → phục vụ
│   │   └─ Không thể thay đổi sau khi đã đặt hàng
│   │
│   └─ Bottom line:
│       └─ Waterfall = kế hoạch chặt chẽ, ít linh hoạt nhưng ổn định ⭐
│
├─ 3. Agile Methodology (Phương pháp linh hoạt) ⭐
│
│   ├─ Khái niệm:
│   │   ├─ Agile = linh hoạt + thích ứng nhanh
│   │   ├─ Iterative (lặp lại)
│   │   └─ Incremental (từng phần nhỏ)
│   │
│   ├─ Đặc điểm:
│   │   ├─ Tasks chạy song song hoặc theo từng giai đoạn nhỏ
│   │   ├─ Có feedback liên tục từ khách hàng
│   │   ├─ Thay đổi được chấp nhận
│   │   └─ Tập trung vào collaboration (hợp tác)
│   │
│   ├─ Sprint (quan trọng ⭐):
│   │   ├─ Chu kỳ ngắn 1–4 tuần
│   │   └─ Hoàn thành một phần công việc cụ thể
│   │
│   ├─ Khi nào dùng Agile:
│   │   ├─ Yêu cầu chưa rõ ràng từ đầu
│   │   ├─ Khách hàng muốn điều chỉnh liên tục
│   │   ├─ Dự án nhiều rủi ro / thay đổi
│   │   └─ Cần ra sản phẩm nhanh
│   │
│   ├─ Ví dụ:
│   │   ├─ Xây dựng website
│   │   ├─ Làm từng phần: homepage → blog → booking system
│   │   └─ Có thể launch sớm một phần website
│   │
│   └─ Bottom line:
│       └─ Agile = linh hoạt, nhanh, cải tiến liên tục ⭐
│
├─ 4. So sánh Waterfall vs Agile ⭐
│
│   ├─ Waterfall:
│   │   ├─ Linear (tuần tự)
│   │   ├─ Ít thay đổi
│   │   ├─ Kế hoạch từ đầu rất quan trọng
│   │   └─ Phù hợp dự án ổn định
│   │
│   ├─ Agile:
│   │   ├─ Iterative (lặp lại)
│   │   ├─ Linh hoạt cao
│   │   ├─ Feedback liên tục
│   │   └─ Phù hợp môi trường thay đổi
│   │
│   └─ Key difference:
│       └─ Waterfall = predictability | Agile = adaptability ⭐
│
├─ 5. Ý nghĩa thực tế ⭐
│
│   ├─ Không có phương pháp nào “tốt nhất”
│   ├─ PM phải chọn theo:
│   │   ├─ Loại dự án
│   │   ├─ Mức độ thay đổi
│   │   ├─ Customer expectations
│   │   └─ Rủi ro
│   │
│   └─ Thực tế tại Google:
│       └─ Kết hợp nhiều methodology tùy dự án
│
└─ 6. Key Takeaways ⭐
    ├─ Waterfall = kế hoạch cố định, ít thay đổi
    ├─ Agile = linh hoạt, thay đổi liên tục
    ├─ Sprint = đơn vị làm việc trong Agile (1–4 tuần)
    ├─ Agile = mindset, không chỉ là quy trình
    └─ PM giỏi = biết chọn đúng phương pháp cho từng dự án ⭐

#Google Project Management – A Project Manager’s Role in Change Management

```diagram	
A Project Manager’s Role in Change Management
│
├─ 1. Tổng quan ⭐
│   ├─ Change management = quản lý sự thay đổi
│   ├─ Mục tiêu:
│   │   ├─ Đưa dự án hoàn thành vào thực tế
│   │   ├─ Giúp người dùng/nhân viên chấp nhận thay đổi
│   │   └─ Đảm bảo adoption (sự áp dụng) thành công
│   ├─ Quan trọng vì:
│   │   └─ Project chỉ thành công khi được sử dụng đúng cách
│
├─ 2. Tác động của change management 🔄
│
│   ├─ Đối với user (người dùng)
│   │   ├─ Phải học cách sử dụng sản phẩm mới
│   │   └─ Thay đổi thói quen làm việc
│   │
│   ├─ Đối với tổ chức
│   │   ├─ Thay đổi quy trình nội bộ
│   │   ├─ Cập nhật thương hiệu / hệ thống
│   │   └─ Đồng bộ toàn bộ nhân viên
│   │
│   └─ Ví dụ:
│       ├─ Website UI thay đổi → user phải làm quen lại
│       └─ Logo mới → toàn công ty phải cập nhật tài liệu
│
├─ 3. Vai trò của Project Manager trong change management ⭐
│
│   ├─ PM không nhất thiết là người lead change
│   ├─ Nhưng phải:
│   │   ├─ Hỗ trợ quá trình thay đổi
│   │   ├─ Tích hợp change management vào project
│   │   └─ Đảm bảo dự án được “adopt” thành công
│   │
│   └─ Mục tiêu chung:
│       ├─ Project success + Change adoption
│       └─ Giảm resistance (sự chống đối thay đổi)
│
├─ 4. Câu hỏi PM cần đặt ra trước thay đổi ❓
│   ├─ Tổ chức sẽ phản ứng thế nào với thay đổi?
│   ├─ Ai là người ảnh hưởng (influencers)?
│   ├─ Kênh giao tiếp nào hiệu quả nhất?
│   └─ Chiến lược change nào phù hợp dự án?
│
├─ 5. Best practices trong change management ⭐
│
│   ├─ Proactive (chủ động)
│   │   ├─ Thông báo sớm thay đổi
│   │   ├─ Chuẩn bị stakeholder trước
│   │   └─ Giảm bất ngờ
│   │
│   ├─ Tích hợp vào project plan
│   │   ├─ Dành thời gian training
│   │   ├─ Thu thập feedback
│   │   └─ Demo / Q&A / video hướng dẫn
│   │
│   ├─ Communication (giao tiếp liên tục)
│   │   ├─ Cập nhật tiến độ thường xuyên
│   │   ├─ Giải thích lợi ích thay đổi
│   │   └─ Giữ stakeholder luôn được thông tin
│   │
│   ├─ Consistent process (quy trình rõ ràng)
│   │   ├─ Có kế hoạch change management từ đầu
│   │   ├─ Quy định rõ khi nào training / launch
│   │   └─ Giảm rủi ro trễ tiến độ
│   │
│   ├─ Empathy (thấu hiểu)
│   │   ├─ Hiểu cảm xúc khi có thay đổi
│   │   ├─ Hỗ trợ team vượt qua khó khăn
│   │   └─ Giảm resistance
│   │
│   └─ Tools hỗ trợ 🧰
│       ├─ Survey / feedback forms
│       ├─ Flowcharts (quy trình trực quan)
│       └─ Culture mapping (phân tích văn hóa tổ chức)
│
└─ 6. Key takeaway ⭐
    ├─ Change management = yếu tố quyết định adoption thành công
    ├─ PM cần tích hợp change vào toàn bộ dự án
    ├─ Giao tiếp + empathy + planning là cốt lõi
    └─ Project chỉ thành công khi người dùng thật sự “accept & use” 🚀

#Google Project Management – Corporate and Project Governance

```diagram		
Corporate and Project Governance
│
├─ 1. Tổng quan ⭐
│   ├─ Governance = framework quản lý cách ra quyết định & trách nhiệm
│   ├─ Hiểu đơn giản: "Ai là người chịu trách nhiệm?"
│   ├─ Gồm 2 loại chính:
│   │   ├─ Corporate Governance (cấp công ty)
│   │   └─ Project Governance (cấp dự án)
│   └─ Mục tiêu:
│       ├─ Kiểm soát hoạt động
│       ├─ Đảm bảo hiệu quả
│       └─ Phân quyền rõ ràng
│
├─ 2. Corporate Governance 🏢
│
│   ├─ Định nghĩa
│   │   ├─ Hệ thống tiêu chuẩn & quy tắc của toàn công ty
│   │   ├─ Điều hướng cách tổ chức đạt mục tiêu
│   │   └─ Cân bằng lợi ích:
│   │       ├─ Stakeholders
│   │       ├─ Management
│   │       └─ Customers
│   │
│   ├─ Ảnh hưởng đến:
│   │   ├─ Action plans (kế hoạch hành động)
│   │   ├─ Internal / external controls
│   │   └─ Performance measurement
│   │
│   ├─ Vai trò trong tổ chức
│   │   ├─ Xác định ai có quyền ra quyết định
│   │   ├─ Hỗ trợ quản lý thay đổi (Change Management)
│   │   └─ Đảm bảo tuân thủ & giảm rủi ro
│   │
│   └─ Steering Committee ⭐
│       ├─ Nhóm cố vấn / định hướng chiến lược
│       ├─ Xác định ưu tiên của công ty
│       └─ Hỗ trợ PM trong quyết định quan trọng
│
├─ 3. Project Governance 📊
│
│   ├─ Định nghĩa
│   │   ├─ Framework quản lý quyết định trong dự án
│   │   ├─ Xác định:
│   │   │   ├─ Ai làm gì
│   │   │   └─ Ai chịu trách nhiệm
│   │   └─ Giúp dự án:
│   │       ├─ Đúng tiến độ
│   │       ├─ Đúng ngân sách
│   │       └─ Đúng mục tiêu
│   │
│   ├─ Bao gồm:
│   │   ├─ Policies (chính sách)
│   │   ├─ Processes (quy trình)
│   │   ├─ Procedures (thủ tục)
│   │   └─ Responsibilities (trách nhiệm)
│   │
│   └─ Vai trò PM:
│       ├─ Thiết kế & áp dụng governance cho dự án
│       ├─ Theo dõi & kiểm soát tiến độ
│       └─ Đảm bảo minh bạch quyết định
│
├─ 4. Sự giao thoa Corporate vs Project Governance 🔗
│
│   ├─ Corporate Governance:
│   │   ├─ Quyết định cấp công ty
│   │   └─ Ảnh hưởng toàn tổ chức
│   │
│   ├─ Project Governance:
│   │   ├─ Quyết định cấp dự án
│   │   └─ Tập trung vào delivery
│   │
│   ├─ Liên kết:
│   │   ├─ Align dự án với mục tiêu công ty
│   │   ├─ Cung cấp tài nguyên & hỗ trợ
│   │   ├─ Giảm rủi ro & tăng tuân thủ
│   │   └─ Đảm bảo ra quyết định đúng người đúng cấp
│   │
│   └─ Theo từng giai đoạn:
│       ├─ Initiating: xác định mục tiêu rõ ràng
│       ├─ Planning: phân quyền & trách nhiệm
│       ├─ Executing: học từ sai lầm & điều chỉnh
│       └─ Monitoring: đảm bảo kiểm soát & báo cáo
│
└─ 5. Key takeaway ⭐
    ├─ Governance = nền tảng để quản lý quyết định & trách nhiệm
    ├─ Corporate governance = cấp công ty
    ├─ Project governance = cấp dự án
    ├─ PM cần hiểu rõ để:
    │   ├─ Biết ai quyết định gì
    │   ├─ Tránh rủi ro
    │   └─ Làm việc hiệu quả hơn
    └─ Hiểu governance = lợi thế lớn khi làm PM
