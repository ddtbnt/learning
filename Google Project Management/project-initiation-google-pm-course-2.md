#Google Project Management – Project Initiation: Starting a Successful Project - Performing a Cost-Benefit Analysis

```diagram	
Performing a Cost-Benefit Analysis
│
├─ 1. Tổng quan ⭐
│   ├─ Cost-Benefit Analysis (CBA) = so sánh lợi ích vs chi phí dự án
│   ├─ Mục tiêu:
│   │   ├─ Quyết định có nên làm dự án hay không
│   │   ├─ Tối đa hóa lợi ích
│   │   └─ Giảm rủi ro & lãng phí
│   ├─ Dựa trên dữ liệu khách quan (objective data)
│   └─ Giúp tránh bias từ stakeholder
│
├─ 2. Lợi ích của Cost-Benefit Analysis 📊
│
│   ├─ Ra quyết định tốt hơn
│   │   └─ Chọn dự án có giá trị cao nhất
│   │
│   ├─ Giao tiếp rõ ràng với stakeholder
│   │   └─ Dễ thuyết phục leadership
│   │
│   ├─ Giảm rủi ro
│   │   └─ Nhìn trước chi phí & lợi ích
│   │
│   └─ Tối ưu tài nguyên
│       └─ Tránh đầu tư sai dự án
│
├─ 3. Câu hỏi khi phân tích CBA ❓
│
│   ├─ Lợi ích (Benefits)
│   │   ├─ Dự án tạo ra giá trị gì?
│   │   ├─ Tiết kiệm bao nhiêu tiền?
│   │   ├─ Tăng doanh thu bao nhiêu?
│   │   ├─ Tiết kiệm thời gian không?
│   │   └─ Cải thiện customer experience không?
│   │
│   ├─ Chi phí (Costs)
│   │   ├─ Chi phí ban đầu (upfront cost)
│   │   ├─ Chi phí vận hành (ongoing cost)
│   │   ├─ Chi phí dài hạn
│   │   └─ Thời gian & nhân lực bỏ ra
│   │
│   └─ Chi phí & lợi ích vô hình (Intangible)
│       ├─ Customer satisfaction
│       ├─ Employee morale
│       ├─ Productivity
│       └─ Brand perception
│
├─ 4. Chi phí & lợi ích vô hình ⚠️
│
│   ├─ Intangible benefits:
│   │   ├─ Khó đo bằng tiền
│   │   └─ Nhưng ảnh hưởng lớn đến doanh nghiệp
│   │
│   ├─ Intangible costs:
│   │   ├─ Giảm hài lòng khách hàng
│   │   ├─ Giảm morale nhân viên
│   │   └─ Ảnh hưởng thương hiệu
│   │
│   └─ Cách ước lượng:
│       ├─ Dựa trên dự án cũ
│       ├─ Nghiên cứu thị trường
│       └─ Tham khảo chuyên gia
│
├─ 5. ROI (Return on Investment) 💰
│
│   ├─ Ý nghĩa:
│   │   └─ Đo mức độ lợi nhuận của dự án
│   │
│   ├─ Công thức:
│   │   └─ ROI = (G - C) ÷ C × 100%
│   │
│   ├─ Trong đó:
│   │   ├─ G = Gain (lợi ích tài chính)
│   │   └─ C = Cost (chi phí)
│   │
│   └─ Quy tắc:
│       ├─ ROI > 10% → thường là tốt
│       └─ ROI càng cao → dự án càng hấp dẫn
│
├─ 6. Ví dụ tính ROI 🧮
│
│   ├─ Chi phí:
│   │   ├─ Upfront: $6,000
│   │   ├─ Monthly: $25 × 12 = $300
│   │   └─ Total C = $6,300
│   │
│   ├─ Lợi ích:
│   │   └─ G = $10,000
│   │
│   ├─ Tính toán:
│   │   ├─ 10,000 - 6,300 = 3,700
│   │   ├─ 3,700 ÷ 6,300 = 0.5873
│   │   └─ ROI = 58.7%
│   │
│   └─ Kết luận:
│       └─ ROI cao → nên thực hiện dự án
│
└─ 7. Key takeaway ⭐
    ├─ CBA giúp quyết định dự án có đáng làm không
    ├─ So sánh lợi ích vs chi phí (cả hữu hình & vô hình)
    ├─ ROI là công cụ quan trọng để đo hiệu quả
    └─ ROI cao = dự án có giá trị kinh tế tốt
	
```

#Google Project Management – Project Initiation: Starting a Successful Project - SMART Goals: Making Goals Meaningful

```diagram	
SMART Goals: Making Goals Meaningful
│
├─ 1. Tổng quan ⭐
│   ├─ SMART Goals = cách đặt mục tiêu rõ ràng và đo lường được
│   ├─ Áp dụng trong quản lý dự án để:
│   │   ├─ Hiểu phạm vi công việc (Scope)
│   │   ├─ Đánh giá tiến độ & thành công
│   │   └─ Giảm mơ hồ giữa các stakeholder
│   └─ PM không nhất thiết tự đặt mục tiêu, nhưng phải hiểu & làm rõ mục tiêu
│
├─ 2. SMART là gì? ⭐
│
│   ├─ S – Specific (Cụ thể)
│   │   ├─ Mục tiêu rõ ràng, không mơ hồ
│   │   └─ Team không hiểu sai yêu cầu
│   │
│   ├─ M – Measurable (Đo lường được)
│   │   ├─ Có chỉ số cụ thể (metrics)
│   │   ├─ Ví dụ:
│   │   │   ├─ Số lượng feature hoàn thành
│   │   │   └─ Doanh thu, chi phí, số lỗi
│   │   └─ Giúp xác định “đạt hay chưa đạt”
│   │
│   ├─ A – Attainable (Khả thi)
│   │   ├─ Mục tiêu thực tế, có thể đạt được
│   │   └─ Team đồng ý với tính khả thi
│   │
│   ├─ R – Relevant (Liên quan)
│   │   ├─ Phù hợp với chiến lược công ty
│   │   └─ Hỗ trợ project charter & mục tiêu tổ chức
│   │
│   └─ T – Time-bound (Có thời hạn)
│       ├─ Có deadline rõ ràng
│       └─ Giúp kiểm soát tiến độ dự án
│
├─ 3. Điểm quan trọng: Measurable (M) ⭐
│   ├─ Là yếu tố quan trọng nhất để đánh giá hiệu quả
│   ├─ Giúp tránh mơ hồ trong kỳ vọng
│   ├─ Chỉ số phải có ý nghĩa (không phải số nào cũng hữu ích)
│   └─ Ví dụ:
│       ├─ Không tốt: số lần họp của engineer
│       └─ Tốt hơn: số feature hoàn thành / số bug xử lý
│
├─ 4. Ví dụ SMART Goal ⭐
│
│   ├─ Mục tiêu ban đầu:
│   │   └─ Hoàn thành Google Career Certificate
│   │
│   ├─ Phân tích SMART:
│   │   ├─ Specific: Hoàn thành certificate
│   │   ├─ Measurable: Có chứng chỉ hoàn thành
│   │   ├─ Attainable: 1 course/tháng
│   │   ├─ Relevant: Hỗ trợ đổi nghề
│   │   └─ Time-bound: 6 tháng
│   │
│   └─ SMART Goal hoàn chỉnh:
│       └─ Hoàn thành Google Career Certificate trong 6 tháng, mỗi tháng 1 khóa học
│
└─ 5. Key takeaway ⭐
    ├─ SMART goals giúp biến mục tiêu mơ hồ thành mục tiêu rõ ràng
    ├─ Giúp PM theo dõi tiến độ & đánh giá thành công
    └─ Là công cụ quan trọng để quản lý phạm vi & kỳ vọng dự án

```

```diagram		
│

│   │   ├─ Specific
│   │   │   └─ Update website + marketing & sales strategy

│   │       └─ Deadline: cuối năm

│   │   ├─ Chỉ nói “tăng retention”

│   │   ├─ Không có thời hạn
│   │   └─ Thiếu logic đo lường
│
│   ├─ SMART Goal hoàn chỉnh
│   │   └─ “Office Green will raise their overall customer retention rate by 10% by the end of the year by implementing a new Operations & Training plan for the Plant Pals service.”

│   ├─ Phân tích SMART:
│   │
│   │   ├─ Specific
│   │   │   └─ Implement Operations & Training plan cho Plant Pals
│   │   │
│   │   ├─ Measurable
│   │   │   └─ +10% customer retention rate
│   │   │
│   │   ├─ Attainable
│   │   │   └─ Có nhu cầu khách hàng + khả năng cải thiện dịch vụ
│   │   │
│   │   ├─ Relevant
│   │   │   └─ Giữ khách hàng → tăng doanh thu tổng thể
│   │   │
│   │   └─ Time-bound
│   │       └─ Deadline: cuối năm
```

#Google Project Management – Project Initiation: Starting a Successful Project - Creating OKRs for Your Project

```diagram		
Creating OKRs for Your Project
│
├─ 1. Tổng quan ⭐
│   ├─ OKR = Objectives and Key Results
│   ├─ Dùng để:
│   │   ├─ Định nghĩa mục tiêu (What)
│   │   └─ Đo lường kết quả (How to know it is achieved)
│   ├─ Áp dụng cho:
│   │   ├─ Organization-level OKRs
│   │   └─ Project-level OKRs
│   └─ PM dùng OKR để:

│       ├─ Đồng bộ mục tiêu team
│       └─ Tạo động lực cho team
│
├─ 2. OKR là gì? ⭐
│
│   ├─ Objectives (Mục tiêu)
│   │   ├─ Mô tả “cái cần đạt được”
│   │   ├─ Mang tính định hướng & truyền cảm hứng
│   │   └─ Không cần số liệu cụ thể
│   │
│   ├─ Key Results (Kết quả then chốt)
│   │   ├─ Đo lường được bằng số liệu
│   │   ├─ Xác định khi nào đạt mục tiêu
│   │   └─ Có deadline rõ ràng
│
├─ 3. Cách xây dựng Objectives ⭐
│
│   ├─ Đặc điểm của Objective tốt:
│   │   ├─ Aspirational (tham vọng)
│   │   ├─ Aligned (phù hợp mục tiêu công ty)
│   │   ├─ Action-oriented (hướng hành động)
│   │   ├─ Concrete (rõ ràng)
│   │   └─ Significant (có tác động lớn)
│
│   ├─ Ví dụ Objectives:
│   │   ├─ Build secure data security software
│   │   ├─ Improve web analytics & conversions
│   │   ├─ Provide top-performing service
│   │   └─ Increase market reach
│
│   ├─ Câu hỏi kiểm tra:
│   │   ├─ Có giúp đạt mục tiêu dự án không?
│   │   ├─ Có phù hợp OKR công ty không?
│   │   ├─ Có truyền cảm hứng không?
│   │   └─ Có tạo impact lớn không?
│
├─ 4. Cách xây dựng Key Results ⭐
│
│   ├─ Mỗi Objective → 2–3 Key Results
│   ├─ Đặc điểm Key Results tốt:
│   │   ├─ Result-oriented (kết quả, không phải task)
│   │   ├─ Measurable (đo được)
│   │   ├─ Time-bound (có thời hạn)
│   │   └─ Aggressive nhưng realistic
│
│   ├─ Ví dụ Key Results:
│   │   ├─ +X% new signups in first quarter
│   │   ├─ +X% advertiser spend
│   │   ├─ X% feature adoption rate
│   │   ├─ ≤2 critical bugs per sprint
│   │   └─ Maintain unsubscribe rate ≤ X%
│
│   ├─ Câu hỏi kiểm tra:
│   │   ├─ Success nghĩa là gì?
│   │   └─ Metric nào chứng minh đạt mục tiêu?
│
├─ 5. Best Practices OKR ⭐
│
│   ├─ Objective = Inspirational (truyền cảm hứng)
│   ├─ Key Results = Tactical (cụ thể & đo lường)
│   ├─ 1 Objective → 2–3 Key Results
│   ├─ Ghi lại OKRs trong project plan
│   └─ Link OKRs với tiến độ dự án
│
├─ 6. OKR vs SMART Goals ⭐
│
│   ├─ SMART Goals:
│   │   ├─ Tập trung vào 1 mục tiêu cụ thể
│   │   ├─ Đầy đủ 5 yếu tố SMART
│   │   └─ Dễ dùng cho cá nhân / task nhỏ
│   │
│   ├─ OKR:
│   │   ├─ Dùng cho team / tổ chức
│   │   ├─ Mang tính chiến lược hơn
│   │   ├─ Có nhiều Key Results
│   │   └─ Khuyến khích “stretch goals”
│
└─ 7. Key takeaway ⭐
    ├─ Objectives = What you want to achieve
    ├─ Key Results = How you measure success
    ├─ OKRs giúp PM:
    │   ├─ Rõ scope
    │   ├─ Đo hiệu quả
    │   └─ Đồng bộ team
    └─ Là công cụ quan trọng trong project planning

```

```diagram	
│   │   └─ Có cần giấy phép / thiết kế không?
```

#Google Project Management – Project Initiation: Starting a Successful Project - RACI CHART – PHÂN CÔNG TRÁCH NHIỆM TRONG PROJECT

```diagram	
RACI CHART – PHÂN CÔNG TRÁCH NHIỆM TRONG PROJECT
│
├── 1. MỤC ĐÍCH
│   ├── Xác định rõ vai trò từng người

│   └── I = Informed (Người được thông báo)

│   ├── Người chịu trách nhiệm cuối cùng
│   ├── Chỉ 1 người duy nhất / task
│   ├── Quyết định “done hay chưa”

│       ├── Ai phê duyệt?
│       └── Ai chịu trách nhiệm nếu fail?
│
├── 5. CONSULTED (C)
│   ├── Người đưa ý kiến / feedback
│   ├── Giao tiếp 2 chiều (two-way communication)

│   └── Câu hỏi:
│       ├── Ai có kiến thức liên quan?
│       ├── Ai bị ảnh hưởng bởi task?
│       └── Ai cần góp ý?
│
├── 6. INFORMED (I)
│   ├── Người chỉ cần cập nhật thông tin
│   ├── Không cần phản hồi
│   ├── Giao tiếp 1 chiều (one-way)
│   └── Câu hỏi:
│       ├── Ai cần biết kết quả?
│       └── Ai bị ảnh hưởng sau khi xong task?
│
├── 7. VÍ DỤ (OFFICE GREEN)
│
│   ├── Task: Xác định giá sản phẩm
│   │
│   ├── R (Responsible)
│   │   └── Financial Analyst (phân tích giá)
│   │
```
